1. Application Overview

MAAFlo (Marketing Automation Analytics Flow) is a local-only, single-user desktop web application built for individual advertisers and small businesses to manage, track, and analyze their own Google Ads campaigns from a unified local dashboard.

The application runs entirely on the user's own computer (localhost:3001) and does not collect, transmit, or store any user data on external servers. All data is stored in a local SQLite database on the user's machine.

Key point: MAAFlo is not a SaaS platform, ad agency tool, or multi-tenant service. It is a personal productivity tool used exclusively to manage the operator's own Google Ads accounts.

2. Purpose & Use of the Google Ads API

MAAFlo integrates the Google Ads API to allow the operator to create and manage campaigns in their own accounts without switching between multiple interfaces. The API is used strictly for first-party campaign management — no third-party account access is involved.

Core API Use Cases

FeatureAPI Service UsedPurpose

Account Discovery CustomerService.listAccessibleCustomers List accessible accounts linked to the authenticated Google account
OAuth Authentication Google OAuth 2.0 Securely authenticate the operator via their own Google account

3. Technical Architecture

Stack

  • Frontend: React 18 + TypeScript, runs in browser at localhost:5173
  • Backend: Node.js + Express API server at localhost:3001
  • Database: SQLite (local file, data/maaflo.db)
  • Authentication: Google OAuth 2.0 with offline access (refresh token stored locally)

API Call Flow

  1. User authenticates via Google OAuth 2.0 — refresh token stored in local SQLite DB
  2. On campaign push, the Node.js server exchanges the refresh token for an access token
  3. The server calls Google Ads API v20 with the access token and developer token
  4. A CampaignBudget is created, then a Campaign resource is created referencing that budget
  5. The returned Google Ads Campaign ID is saved locally for reference

API Endpoints Used

EndpointMethodFrequency
/customers:listAccessibleCustomers GET Once at setup / on-demand
/customers/{id}/campaignBudgets:mutate POST On campaign creation (operator-initiated)
/customers/{id}/campaigns:mutate POST On campaign creation (operator-initiated)

4. Data Handling & Privacy

  • All Google Ads data (campaign IDs, tokens) is stored only in the local SQLite database on the operator's own machine
  • No data is sent to third-party servers — the only outbound connections are to Google's own APIs
  • OAuth refresh tokens are stored locally and never transmitted outside the user's machine
  • The application has no login system, user accounts, or cloud storage
  • MAAFlo accesses only the accounts belonging to the authenticated Google user

5. Why Basic Access is Required

The Google Ads API test developer token only allows access to test accounts. MAAFlo needs to create and manage campaigns in the operator's real production Google Ads account. Basic Access is the minimum access level required to interact with live accounts, which is the entire purpose of the integration.

MAAFlo does not require Standard or higher access — the application does not manage third-party accounts, does not perform bulk data exports, and does not make automated/scheduled API calls. All API calls are triggered manually by the operator through the UI.

6. Compliance & Policy

  • API usage complies with Google Ads API Best Practices
  • Campaigns are always created with status PAUSED — the operator activates them manually in Google Ads
  • No automated bidding manipulation or policy-violating campaign types are used
  • The application is used solely by the developer/operator for their own advertising accounts
